# Plan 0010: narinfo text format parser and generator

## Context

narinfo is the binary-cache metadata format ‚Äî the thing at `https://cache.nixos.org/<hashpart>.narinfo` that tells a substituter where to find the NAR, what it hashes to, what paths it references, and who signed it. Phase 1b doesn't have a binary-cache HTTP server (that's phase 2a), but the phase 1b task list includes narinfo as a prerequisite: the `PathInfo` wire type (P0000) and the narinfo text format share the same semantic fields, and phase 2a's `rio-store` filesystem backend serializes `PathInfo` as narinfo text on disk.

The format is line-based `Key: Value` pairs, no quoting, no escaping, `\n`-terminated. Simple enough that it's tempting to regex-parse, but there's enough structure (required fields, fields that appear exactly once vs. `Sig:` which can repeat, reference fields that are space-separated basenames) that a real parser catches more bugs.

## Design

The parser iterates lines, splits each on the first `:`, trims whitespace, and dispatches on the key. Single-value fields (`StorePath`, `URL`, `Compression`, `NarHash`, `NarSize`, `Deriver`, `CA`, `FileHash`, `FileSize`) reject duplicates ‚Äî a second `NarHash:` line is a parse error. Multi-value fields (`Sig:`) accumulate into a `Vec`. `References:` is single-occurrence but space-separated within the line.

Required fields are `StorePath`, `URL`, `Compression`, `NarHash`. Missing any of those is a parse error. Everything else is optional. This version encoded optional-absent as empty string (`deriver: String` defaulting to `""`); P0017 changed them to `Option<String>` so absence is semantically distinct from present-but-empty.

The `NarInfoBuilder` is the serializer. It's a typestate-lite pattern ‚Äî you chain `.store_path(...)`, `.url(...)`, etc., then call `.build()`, which emits the text format. Field ordering in the output follows Nix's convention (the order `cache.nixos.org` uses), so a narinfo generated by rio serializes the same as one generated by Nix.

## Outcome

Parser handles all standard fields with duplicate detection. Builder produces Nix-compatible field ordering. Not used anywhere in phase 1b ‚Äî it sits dormant until phase 2a's filesystem store backend. But landing it alongside the ATerm and NAR parsers meant all three format foundations were reviewed in the same pass.
