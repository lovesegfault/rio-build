## Quick Start Commands

This is a Rust workspace project managed with Nix flakes.

```bash
# Enter development environment (using direnv, already configured)
direnv allow

# Or manually enter the Nix development shell
nix develop

# IMPORTANT: All cargo commands should be run within nix develop environment
# This ensures proper dependencies (protoc, libclang, etc.) are available

# Build all workspace members
nix develop -c cargo build

# Build specific binary
nix develop -c cargo build -p rio-build
nix develop -c cargo build -p rio-agent

# Run tests
nix develop -c cargo test

# Run a specific test
nix develop -c cargo test test_name

# Check code without building
nix develop -c cargo check

# Run clippy linter
nix develop -c cargo clippy

# Format code (all files in project)
nix develop -c treefmt

# Or format just Rust code
nix develop -c cargo fmt

# Watch for changes and rebuild
nix develop -c cargo watch -x check

# Run the CLI client
nix develop -c cargo run -p rio-build

# Run an agent node
nix develop -c cargo run -p rio-agent
```

## Development Best Practices

### Dependencies

**Always use workspace dependencies!** All crate versions are defined in root `Cargo.toml` `[workspace.dependencies]`. In crate-specific Cargo.toml files, use:
```toml
dependency-name = { workspace = true }
```

**Keep dependencies up to date**: Run `cargo update --verbose` periodically to check for updates.

### Error Handling

**Never use `.unwrap()` in production code.** Use `.expect()` only when the invariant is clear and documented.

**Use `.with_context()` for operations that can fail** to provide helpful error messages:
```rust
use anyhow::Context;
use camino::Utf8Path;  // Always use UTF-8 paths

// File operations
tokio::fs::read(path)
    .await
    .with_context(|| format!("Failed to read file: {}", path))?;

// Option types in production code
let agent = state.agents.get(&agent_id)
    .context("agent should exist in cluster state")?;
```

**Use `.expect()` with clear messages for invariants** (when failure indicates a bug):
```rust
// Column family handles (created at DB initialization)
let store = self.db.cf_handle(CF_STORE)
    .expect("CF_STORE column family must exist (created at DB init)");

// Protobuf required fields (protocol violations)
let vote = pb.vote.as_ref()
    .expect("protobuf Vote must have leader_id field");

// Serialization that cannot fail
let json = serde_json::to_vec(&data)
    .expect("RaftCommand serialization cannot fail (all types are Serialize)");
```

**Test functions should return `anyhow::Result<()>`**:
```rust
#[test]
fn test_example() -> anyhow::Result<()> {
    use anyhow::Context;

    let temp_dir = tempfile::tempdir()?;
    let path = Utf8Path::from_path(temp_dir.path())
        .context("temp dir path should be valid UTF-8")?;

    let result = some_operation(path).await?;
    assert_eq!(result, expected);
    Ok(())
}
```

**Benefits of this approach:**
- Production code: Clear error messages when invariants are violated
- Test code: Automatic error propagation with `?`, better failure diagnostics
- No silent failures from `.unwrap()`
- Better debugging with `.context()` providing error chains

### Observability

**Use `#[tracing::instrument]` on all public async functions:**
```rust
#[tracing::instrument(skip(self, large_data), fields(job_id = %job.id))]
pub async fn submit_build(&self, job: BuildRequest) -> JobId {
    // ...
}
```

- Use `skip()` for large/sensitive data
- Extract key identifiers as fields for correlation

### Testing

**Write tests for new functionality** - aim for 70%+ coverage:
```bash
cargo test -p rio-build module::tests
```

## Development Gotchas

**Generated Files**: `.pre-commit-config.yaml` is auto-generated by git-hooks.nix and should not be edited manually. It's already in `.gitignore`.
- Never include Co-Authored-By or other Claude information in commit messages
- Always update TODO.md after completing a phase
