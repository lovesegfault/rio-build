# Glossary

| Term | Definition |
|------|-----------|
| **NAR** | Nix ARchive. A deterministic archive format for serializing store paths. Unlike tar, NAR produces identical output for identical directory trees regardless of filesystem metadata (timestamps, permissions). |
| **Derivation (.drv)** | A build recipe stored in the Nix store. Contains: builder executable, arguments, environment variables, input sources, input derivations, and declared outputs. Serialized in ATerm format. |
| **ATerm** | The serialization format used for `.drv` files. A simple S-expression-like format: `Derive([outputs], [inputDrvs], [inputSrcs], platform, builder, [args], [env])`. |
| **Closure** | The transitive set of all store paths required by a given store path, including itself and all its runtime dependencies. A closure is self-contained: copying it to another machine provides everything needed. |
| **Store path** | A path in `/nix/store/` identified by a hash and a name, e.g. `/nix/store/aaaa...-hello-2.12.1`. The hash encodes the derivation inputs (input-addressed) or output content (content-addressed). |
| **pname** | The "package name" component of a derivation, read from the derivation's environment variables (`env.pname`, falling back to `env.name`). Used for build duration estimation matching. |
| **CAS** | Content-Addressable Store. A storage system where objects are identified by the hash of their content. Enables deduplication: identical content is stored once regardless of how many paths reference it. |
| **CA** | Content-Addressed (derivation). A derivation whose output store path is determined by the *content* of the output, not by its inputs. Enables early cutoff: if rebuilding produces the same output, downstream builds can be skipped. |
| **Modular derivation hash** | The hash computed by `hashDerivationModulo` in Nix. For input-addressed derivations, this includes the derivation's full inputs. For CA derivations, it excludes output paths and depends only on the derivation's fixed attributes. Used for deduplication in DAG merging. |
| **IFD** | Import-From-Derivation. When Nix evaluation depends on a build result (e.g., `import (pkgs.runCommand ...)`). The evaluator blocks until the build completes, creating a tight eval-build dependency cycle. |
| **FOD** | Fixed-Output Derivation. A derivation with a pre-declared output hash (e.g., `fetchurl`). FODs are allowed network access during build and are always content-addressed. |
| **DAG** | Directed Acyclic Graph. The dependency structure of derivations --- each derivation depends on its inputs, forming a graph with no cycles. rio-build's scheduler operates on the global DAG of all concurrent builds. |
| **narinfo** | A text file describing a store path's metadata: store path, NAR hash, NAR size, references, deriver, signatures. Served by binary caches at `/<hash>.narinfo`. |
| **nixbase32** | Nix's custom base32 encoding (characters: `0123456789abcdfghijklmnpqrsvwxyz` --- note missing `e`, `o`, `t`, `u`). Used in store path hashes. |
| **FastCDC** | Fast Content-Defined Chunking. An algorithm that splits byte streams into variable-size chunks at content-dependent boundaries. Enables deduplication even when data shifts within a file. |
| **BLAKE3** | A fast cryptographic hash function used internally by rio-build for chunk content addressing. Not used for Nix-facing hashes (those use SHA-256). |
| **Worker protocol** | The Nix daemon's RPC protocol, spoken over SSH channels. Uses little-endian u64 integers, padded strings, and a STDERR streaming loop for progress/log reporting. |
| **STDERR loop** | The Nix protocol's mechanism for streaming progress, logs, and errors during an operation. The server sends STDERR_NEXT, STDERR_START_ACTIVITY, etc. messages until STDERR_LAST signals the operation result follows. |
| **Build hook** | Nix's mechanism for delegating builds to remote machines via `--builders`. The local daemon invokes a hook program that connects to the remote builder. rio-build supports this mode but with reduced scheduling optimization (no DAG visibility). |
| **FUSE** | Filesystem in Userspace. A Linux kernel interface that allows implementing filesystems in user-space programs. rio-build uses FUSE (via the `fuser` crate) to present a lazy-fetching store view at `/var/rio/fuse-store`, backed by remote content from rio-store. |
| **overlayfs** | A Linux union filesystem that layers a writable "upper" directory over one or more read-only "lower" directories. rio-build uses stacked lowers (`/nix/store:{fuse_mount}` --- host store first, FUSE second) so nix-daemon and its dependencies remain reachable after the overlay's merged dir is bind-mounted at `/nix/store` in the child's mount namespace. |
| **CRD** | Custom Resource Definition. A Kubernetes extension mechanism for defining custom API objects. rio-build defines `BuilderPool`, `BuilderPoolSet`, `FetcherPool`, and `ComponentScaler` CRDs managed by the rio-controller operator. |
| **mTLS** | Mutual TLS. Both client and server authenticate each other with certificates. All internal gRPC uses mTLS via cert-manager-issued per-component certificates (`RIO_TLS__CERT_PATH`/`KEY_PATH`/`CA_PATH`). |
| **IRSA** | IAM Roles for Service Accounts. An EKS feature that maps Kubernetes ServiceAccounts to AWS IAM roles, allowing pods to assume IAM roles without static credentials. Used for S3 access in rio-store. |
| **IMDSv2** | Instance Metadata Service version 2. AWS's token-based metadata endpoint. Setting hop limit=1 on worker nodes prevents containers from accessing node metadata, a defense-in-depth measure. |
| **EMA** | Exponential Moving Average. A weighted average that gives more weight to recent observations. Used by the scheduler for build duration estimation (alpha=0.3). |
| **seccomp** | Secure Computing Mode. A Linux kernel feature that restricts which system calls a process can make. The controller sets `BuilderPoolSpec.seccompProfile` (default `Localhost` profile denying `ptrace`/`bpf`/`setns`/`process_vm_*`) plus granular capabilities (`SYS_ADMIN`, `SYS_CHROOT`) instead of `privileged: true`. |
| **NetworkPolicy** | A Kubernetes resource that controls network traffic between pods. rio-build uses NetworkPolicies to restrict worker egress to only the scheduler and store, blocking access to the Kubernetes API and cloud metadata services. |
| **PDB** | PodDisruptionBudget. A Kubernetes resource that limits how many pods can be simultaneously unavailable during voluntary disruptions (e.g., node drain). rio-build creates PDBs to maintain minimum build capacity. |
| **NLB** | Network Load Balancer. An AWS L4 load balancer used for the gateway's SSH ingress. Configured with extended idle timeout (3600s) to support long-running build sessions. |
| **HPA** | Horizontal Pod Autoscaler. A Kubernetes built-in autoscaling mechanism. rio-build does NOT use HPA for builders --- builder pods are one-shot Jobs spawned/reaped by rio-controller based on scheduler queue depth. |
| **Write-ahead manifest** | rio-store's pattern for durable writes: chunk references are written to a pending manifest before uploading chunks, then promoted to committed after all chunks are verified. Protects against orphaned chunks and broken manifests. |
| **Inline storage** | A fast-path in rio-store for NARs below 256 KiB (`INLINE_THRESHOLD`, compile-time const) that bypasses FastCDC chunking. Stored directly in the PostgreSQL `manifests.inline_blob` BYTEA column --- inline blobs **never touch S3**. |
| **Build (request)** | A full build request (DAG of derivations) submitted via `SubmitBuild`. Contains one or more derivations. Tracked by `build_id`. |
| **Build (derivation)** | Execution of a single derivation on a worker. Multiple derivation builds comprise a build request. Tracked by `derivation_hash`. |
| **Blob** | An opaque binary object. In rio-store: chunks (FastCDC pieces keyed by BLAKE3) live in S3; inline NARs (< 256 KiB) live in a PostgreSQL BYTEA column, not S3. |
| **DerivedPath** | A Nix type representing either a plain store path (Opaque) or a derivation output reference (Built). Format: `drvPath!output1,output2` or `drvPath!*` for all outputs. |
| **Singleflight** | A concurrency pattern where multiple concurrent requests for the same key share a single in-flight operation. Used in rio-store to avoid duplicate S3 fetches. |
| **Tenant** | An organizational unit (team, project, or user group) that shares a rio-build deployment with data isolation and resource quotas. Identified via SSH key mapping at the gateway. See [Multi-Tenancy](./multi-tenancy.md). |
| **Leader election** | A distributed coordination pattern where one instance is selected as the active leader. rio-build uses Kubernetes Lease objects for scheduler leader election, ensuring only one scheduler instance owns the in-memory DAG at a time. |
| **GC root** | A store path that is protected from garbage collection. In rio-build, GC roots include outputs of active/queued builds, paths referenced by `wopAddTempRoot`, and paths within the configured retention period. |
| **Assignment token** | An HMAC-SHA256-signed token issued by the scheduler when dispatching work to an executor. Claims: `(executor_id, drv_hash, expected_outputs, is_ca, expiry_unix)`. Verified by rio-store on `PutPath`. See [Security](./security.md). |
| **Prefetch hint** | A message sent by the scheduler to a worker via the `BuildExecution` stream before assigning a build, listing input closure paths that the worker's FUSE cache should pre-warm. Converts serial "fetch then build" into overlapped execution. |
| **Poison derivation** | A derivation that has consistently failed on multiple different workers (default threshold: 3 distinct workers). Marked as poisoned to prevent infinite retry loops. Auto-expires after a configurable TTL (default: 24h). See [Error Taxonomy](./errors.md). |
| **DAG actor** | The single-owner Tokio task in the scheduler that owns the in-memory global DAG. All DAG mutations (merges, completions, cancellations) are processed sequentially via an `mpsc` channel, eliminating lock contention. See [rio-scheduler](./components/scheduler.md). |
| **rio-fuse** | The custom FUSE filesystem implemented as a module (`fuse/`) inside `rio-builder` (not a separate crate or binary). Mounts at `/var/rio/fuse-store` and lazily fetches store path content from rio-store via gRPC on demand, caching on local SSD. |
| **Temp root** | A connection-scoped temporary GC root registered via `wopAddTempRoot`. Prevents GC of store paths that a client is actively using during an SSH session. Lost on gateway restart; the store's 24-hour GC grace period provides safety. |
| **Backpressure** | A flow control mechanism where a downstream component signals upstream to slow down. In rio-build, the scheduler applies backpressure via gRPC flow control windows and bounded actor queue depth. See [rio-scheduler backpressure](./components/scheduler.md). |
| **SITA-E** | Size Interval Task Assignment with Equal load. A task assignment algorithm by Harchol-Balter that partitions jobs into size classes and sets cutoff boundaries to equalize load across hosts. rio-build adapts SITA-E for scheduling derivations to right-sized worker pools. See [ADR-015](./decisions/015-size-class-routing.md). |
| **TAGS** | Task Assignment by Guessing Size. An algorithm for routing jobs when sizes are unknown, using duration estimates with kill/restart for misclassified jobs. rio-build uses the estimation concept but replaces kill/restart with passive misclassification tracking (Nix builds are non-preemptible). |
| **Size class** | A category of builder pool sized for a specific range of build durations and resource needs (e.g., small/medium/large). Defined via `[[size_classes]]` in `scheduler.toml` or the `BuilderPoolSet` CRD. The scheduler routes derivations to the matching class based on estimated duration. |
| **Misclassification** | When a derivation routed to a size class exceeds 2x the class's duration cutoff, indicating the duration estimate was too low. The scheduler updates the EMA with a penalty and routes future instances to a larger class. |
| **BuilderPoolSet** | A CRD defining multiple size-class builder pools with per-class resource allocations and autoscaling. The controller maintains one child `BuilderPool` per class. See [rio-controller](./components/controller.md#builderpoolset). |
